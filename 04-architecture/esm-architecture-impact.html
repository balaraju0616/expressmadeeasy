ESM Architecture Impact

Why ES Modules affect every import, export, and route connection

What ESM Means Here

The entire KeshavSoft ecosystem uses ES Modules (type: module in package.json). This means every import must use the full file path including .js extension, and every module must explicitly export what it exposes.

// package.json
{ "type": "module" }

// Every import needs .js
import { router } from './Api/routes.js'; // correct
import { router } from './Api/routes';    // fails

Named vs Default Exports

The most common runtime error: app.js expects a default export but routes.js uses a named export. ESM does not silently coerce these.

// routes.js — named export
export { router };

// app.js — must use named import
import { router } from './Api/routes.js';

// Wrong — default import of named export
import router from './Api/routes.js'; // Error

How This Affects Generators

Auto-generated files follow the named export convention throughout. If you manually edit a file and switch to default exports, it will break every file that imports from it.

// Generated end-points.js
export { router };

// Generated V1/routes.js
import { router as journalsRouter } from './journals/end-points.js';
